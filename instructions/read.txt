5) LOGIN / AUTH PATTERN (MUST BE CONSISTENT)
For each page that supports login (index, AccountHistory, OCMHome, OCMUser, Admin, etc.):
- The page-level Google login UI normally uses:
 - Container: `googleLoginArea`
 - Primary button container: `googleBtn`
 - Fallback button: `fallbackBtn` (calls `startFallbackLogin`)
 - Status text: `loginStatus`

Expected flow:
1) Receive a Google `id_token` (GSI popup or redirect fallback).
2) Verify with: `https://oauth2.googleapis.com/tokeninfo?id_token=...`
3) Call backend: `WEB_APP_URL?action=me&idToken=...`
4) Normalize user so `user.playerName` and `user.email` are usable.
5) Save token via `saveIdToken` (auth-storage.js)
6) Call `topbarSetAuthState({ idToken, user, isAdmin, balanceBT })`
7) Trigger page-specific loads.

Restore on refresh:
- Use `initAuthFromStorage()` and then call the same page-level helper used by normal login
 (example: `applyAuthFromToken(idToken)` or `onLogin(idToken, true)`).

6) TOP BAR BEHAVIOR (ALWAYS USE SHARED topbar.js)
- DO NOT create custom `#topBar` markup in individual HTML pages.
- Always rely on `topbar.js` to:
 - Inject the shared top bar when `window.initSharedTopBar()` is called.
 - Manage nav buttons and Login/Logout visibility.
 - Display balance and `user.playerName`/`user.email`.

Pages MUST:
- Call `window.initSharedTopBar()` once in `window.onload` (or equivalent)
- Add class `withTopBar` to `<body>` so content isn’t hidden
- Call `window.topbarSetAuthState(...)` whenever:
 - Login completes
 - Balance changes
 - Logout happens

7) CURRENT SHARED CSS + FOX/PAPER BACKGROUND SYSTEM (HOW IT WORKS)
Global styling is layered and intentionally split:

A) Layout baseline (structure/spacing)
- `css/site-layout.css`
 - Provides site-wide layout rules: spacing, section width, common typography sizing,
 and the `withTopBar` body offset pattern.
 - Most pages include this so the UI “feels” consistent.

B) Theme + decorations (background/visual identity)
- `css/site-theme.css`
 - Defines the site-wide theme tokens (colors, fonts, “parchment” look rules)
 - Provides base background variables/classes used by decor/paper panels.
 - NOTE: Some pages might not include it yet; check the page `<head>`.

- `css/panels-paper.css`
 - Implements “paper/panel” surfaces:
 - Paper-like cards, bordered panels, and paper texture usage.
 - Pages typically wrap content in `.section` / “panel” structures that pick up these styles.

C) JS-driven decorative elements
- `shared/site-decor.js`
 - Adds/positions decorative assets (foxes and similar) and/or toggles classes based on page.
 - The fox/paper visuals are a combination of:
 - CSS background layers (paper texture)
 - Positioned decorative images (foxes) controlled by JS + CSS

D) Asset files (images)
- `images/pack_Ground/paper.png`
- `images/pack_Ground/fox.png`
- `images/pack_Ground/fox_stand.png`

Important behavior notes:
- “Paper background” is generally applied by CSS (`panels-paper.css` + theme rules) to page sections.
- “Fox decorations” are generally added by `shared/site-decor.js` as decorative DOM elements or
 by toggling body classes that CSS uses to show/position fox images.
- Many pages can opt out of paper styling by using a body class like `no-paper`.
- If a page’s background looks “wrong”, verify ALL THREE are included:
 - `css/site-layout.css`
 - `css/panels-paper.css`
 - `shared/site-decor.js`
 And (if used by that page) `css/site-theme.css`.
