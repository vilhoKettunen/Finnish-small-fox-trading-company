1) Project name

Vak’s Store (formerly Finnish Small Fox Trading Company / FSFTC)

2) Project description

What the project is

Vak’s Store is a browser-based trading platform and experimental in-game economy for the Vintage Story server TOPS (The Official Public Server). It supports asynchronous trading (no real-time coordination required) using a ledger currency called EW.

Intended purpose & target audience

  • Audience: TOPS players who want reliable trades without scheduling meetups.
  • Goals: reduce barter friction; avoid Rusty Gear (RG) inflation effects; increase price transparency.

How the project was planned

  • Primarily ad-hoc implementation with daily logs early in the project.
  • A GitHub Issues Kanban board was used later (started ~1 month before March 2026).
  • No separate formal bug list; planned fixes in `txt/plan_fix*` files served as issue documentation.

What was accomplished so far

  • Full end-to-end trading flows: Store requests, OCM listings/trades, account history, and admin toolchain.
  • EW economy mechanics implemented and documented (Whitepaper + Economy Specs).
  • Observability added: Bank Dashboard charts, Price History charts, daily velocity tracking and snapshots.
  • Security hardening and abuse prevention informed by penetration testing and feedback.

Next step(s)

  • Real wipe launch validation: observe real user load, throughput, and EW behavior across the wipe cycle.
  • Complete System Architecture documentation.
  • Automated stock target / price signal improvements.
  • Potential future game integration mechanics.
  • Design versatility improvements (layout/theme options).

Work division among team members

  • Vak — 100% of coding, documentation, and design; operator/admin of the trading platform.
  • Arctic Toast — penetration testing and security feedback only (no links, for privacy).

4) Function description(s) (key functions/features)

Store flow (admin-approved)

  1. Player builds a cart (buy/sell items) on the main store page.
  2. Player submits a request; it goes to a pending admin queue.
  3. Admin approves/denies; on approval EW balances and store stock are updated and a receipt is produced.

OCM flow (player-to-player)

  1. Merchant creates an OCM listing (often peg-based pricing); listing enters review queue.
  2. Admin approves; listing becomes public.
  3. Customer sends a trade request; merchant (or admin) accepts/denies.
  4. On execution: balances are updated, history is written, and (when admin-service applies) the fee is burned.

EW mechanics (issuance / destruction)

  • Issuance: EW is created only when a player sells items to the store.
  • Destruction: EW is removed when players buy from the store, via OCM admin fees, and via insurance metal-withdraw nullification.
  • Public observability: the Bank Dashboard makes macro metrics public (no login needed).

Bank metrics / observability

  • Daily snapshots: EW circulation, wealth distribution, destroyed EW.
  • Trade velocity: per-item daily counters archived by a separate Apps Script project.
  • Public charts on Bank + Price History pages.

Insurance

  • Convert EW to a metals-backed policy via an admin-approved workflow.
  • Supports deposit, withdrawal to EW, and withdrawal of physical metals (nullifies the allocated EW).

Auth / account

  • Google Sign-In (GSI) for obtaining an id_token; backend validates it on every request.
  • Account Settings: player name, mailbox/mall mailbox, leaderboard consent model.
  • Account History: immutable log of all balance changes.

Admin tools

  • Queues: pending store requests, OCM listing review, OCM pending trades, insurance pending actions.
  • Controls: approve/deny, stock adjustments, on-behalf-of operations for support.

5) Issues and solutions

1) Google OAuth / GSI integration complexity

Problem: OAuth token flows, persistence across pages, and correct audience validation created a high learning curve.

Impact: Login reliability issues and additional implementation time.

Root cause / diagnosis: Multi-step OAuth behavior (token issuance, client IDs, audience matching), plus new production/test separation after the domain purchase.

Fix / mitigation: Implemented multi OAuth client ID allow-listing (test vs production). Improved token storage/restore and topbar login state handling.

Result / lessons: Login became more reliable, and the system avoided coupling to a single environment/client ID.

2) “doPost bug” (critical): POST requests not reaching Apps Script reliably

Problem: POST requests stopped reaching the Apps Script backend reliably; the root cause was not determined.

Impact: Write actions became unreliable, blocking core trading operations.

Root cause / diagnosis: Unknown; treated as an external platform/environment failure.

Fix / mitigation: Routed specific write operations through a GET-bypass action routing system (“doGet bypass”) to restore reliability. (Frontend kept a bypass allow-list for write actions so it can send them via GET when needed.)

Result / lessons: Core actions worked reliably again, with a clear contract for which operations are safe/required via GET in this environment.

3) Early Google Sheets linking/reference errors (Jan 2026)

Problem: Reading/writing wrong spreadsheets/tabs/columns during early learning (mis-linked IDs and tab references).

Impact: Incorrect stock updates or missing data until references were corrected.

Root cause / diagnosis: Learning curve around how Apps Script references spreadsheets and tabs; insufficient guardrails initially.

Fix / mitigation: Corrected sheet IDs/tab references and improved column targeting. Later documentation explicitly notes risks like hardcoded column indices.

Result / lessons: Reinforced that Sheets-backed systems must be treated like a schema with migration discipline.

4) OCM request/listing spam prevention (Mar 2026)

Problem: Duplicate trade requests and duplicate listing creation attempts.

Impact: Spammy queues and risk of accidental duplicate actions.

Root cause / diagnosis: Users double-clicking / retrying when network latency makes it unclear whether an action was sent.

Fix / mitigation: Added rate limiting (server-side cache TTL) and frontend “sending…” state/UI prevention to block repeats.

Result / lessons: Fewer accidental duplicates; admin workload reduced.

5) Stats correctness + performance

Problem: Statistics and Top5 calculations initially had correctness issues and performance problems.

Impact: Incorrect charts/leaderboards and slow updates on larger datasets.

Root cause / diagnosis: Per-transaction rebuild patterns and insufficient checkpointing/incremental computation.

Fix / mitigation: Moved stats to scheduled batch jobs; used incremental checkpoints; reduced repeated full scans.

Result / lessons: Faster and more consistent stats updates with clearer computation boundaries.

6) Naming migrations: BT → EW and roles (Customer/Merchant)

Problem: Renaming core concepts risked breaking UI/backend mappings and existing data expectations.

Impact: Bugs such as listing loading issues during transition.

Root cause / diagnosis: Contract drift between stored fields, frontend DTOs, and long-lived legacy identifiers.

Fix / mitigation: Performed systematic renaming, added compatibility shims where needed, and fixed mismatched field access.

Result / lessons: Clearer domain language improved documentation and user understanding, but required careful contract management.

7) Security hardening (penetration testing)

Problem: User input surfaces (profile fields, listing creation) needed abuse resistance.

Impact: Risk of injection-like payloads, malformed records, and abuse via unexpected characters or formats.

Root cause / diagnosis: Any web app with user-supplied inputs needs explicit allow-lists and server-side validation.

Fix / mitigation: Strengthened sanitizers, enforced backend validation, and applied rate limits on sensitive actions.

Result / lessons: Reduced abuse surface; clearer definition of “allowed” formats for user fields.
